FRONTEND EXPLANATION FOR INTERVIEW PREP (UPDATED DARK RAG UI)
Project: React (Vite) frontend for multi-repository RAG GitHub Agent
Date: 2026-04-08

====================================================================
0) OVERVIEW OF WHAT CHANGED
====================================================================

Previous version:
- Basic light dashboard.
- Single in-memory chat list shared for all repositories.
- Plain text chat rendering.

New version:
- Modern dark workspace with simple gray/black color system.
- Repository-centric sidebar with search + active workspace switching.
- Strict per-repository chat persistence in frontend state.
- Markdown rendering with syntax highlighting and code copy button.
- Typing indicator, toasts, smooth repo switch fade, fixed multiline composer.
- Removed starter suggestion chips from empty chat state.
- Neutralized highly vibrant actions (for example Add Repository and Sign Out).
- Sidebar and chat now scroll independently.
- Better accessibility and keyboard behavior.

Why this redesign matters for RAG:
- RAG answers are context-sensitive to repo namespace.
- UI must make context explicit and separated per repository.
- Users need confidence that answers belong to the selected repository workspace.

1) FILE: frontend/src/App.jsx
====================================================================

WHAT (line/block level):
1. Imports route primitives and feature pages.
2. HomeRedirect checks auth and routes to /dashboard or /login.
3. Routes map:
   - /login
   - /signup
   - /logout
   - /dashboard (protected)
4. Wildcard route redirects unknown URLs to HomeRedirect.

WHY:
- Central route map keeps auth navigation deterministic.
- Protected dashboard route prevents unauthenticated API usage from UI layer.

HOW:
- useAuth provides isAuthenticated.
- Navigate performs client-side redirect with replace behavior.

CONTEXT (RAG multi-repo):
- Dashboard is the only place where repo context and chat state live.
- Authentication gate prevents calling protected RAG endpoints without JWT.

CHANGES from previous version:
- Route structure remains stable.
- Works with redesigned dashboard component and new UX logic.

2) FILE: frontend/src/api/client.js
====================================================================

WHAT:
1. Creates one Axios instance with baseURL:
   - VITE_API_BASE_URL or fallback to http://127.0.0.1:8000/api
2. Defines protectedPrefixes:
   - /ingest, /query, /repos
3. Request interceptor:
   - Reads auth_token from localStorage.
   - If URL starts with protected prefix, injects Authorization: Bearer token.

WHY:
- Prevents repeated manual header logic across pages.
- Ensures Supabase RLS protected endpoints always receive JWT.

HOW:
- Axios interceptor runs before each request.
- Token read is dynamic, so latest login/logout state is respected.

CONTEXT:
- /api/query/ and /api/repos/ are repository-sensitive routes.
- Missing token causes 401 and breaks per-user isolation flow.

CHANGES:
- Interceptor logic is retained and remains essential.
- Now used heavily by richer dashboard workflows (refresh, ingest, chat).

3) FILE: frontend/src/context/AuthContext.jsx
====================================================================

WHAT:
1. createContext + provider centralize auth state.
2. token state initializes from localStorage auth_token.
3. login(email, password):
   - POST /auth/login
   - Extracts token from res.data.user (backend-specific response shape)
   - Stores token in localStorage + React state.
4. signup(email, password):
   - POST /auth/signup
   - Calls login immediately after success.
5. logout():
   - Tries POST /auth/logout
   - Always clears local token in finally.
6. useMemo exports stable auth object:
   - token, isAuthenticated, login, signup, logout
7. useAuth hook enforces provider usage.

WHY:
- Auth lifecycle belongs in one source of truth.
- The login response format is unusual; central handling avoids mistakes elsewhere.

HOW:
- Context provider wraps app in main.jsx.
- Components consume auth through useAuth.

CONTEXT:
- Protected RAG routes depend on token propagation.
- Repo list, ingest, and query all fail without this auth chain.

CHANGES:
- Core behavior unchanged.
- Still critical for dark redesign because all new interactions are protected calls.

9) FILE: frontend/src/pages/DashboardPage.jsx (MAIN REDESIGN)
====================================================================

This file contains the largest architectural upgrade.

A) Imports + helper functions
WHAT:
- Adds useRef for auto-scroll and input focus.
- Imports MarkdownMessage component.
- buildMessage helper builds normalized message object:
  - id, role, content, sources, timestamp.
- formatTime helper formats ISO timestamp.

WHY:
- Message normalization makes rendering stable and extensible.
- Timestamp supports chat readability.

HOW:
- Helper functions avoid repeated inline object creation logic.

CONTEXT:
- Message object model is foundation for per-repo chat persistence.

CHANGES:
- New message model introduced.


B) State model
WHAT:
- repos: backend repository metadata list.
- selectedRepo: active workspace.
- repoFilter: sidebar search text.
- showAddRepo: toggles add repository UI in header.
- repoUrl: ingest input value.
- question: composer content.
- messagesByRepo: object map { repoName: message[] }.
- isSwitchingRepo: small transition state.
- isRepoLoading, isIngesting, isAsking: independent async flags.
- toasts: transient notifications.

WHY:
- Multi-repo chat requires strict separation of conversation state.
- Independent loading flags prevent blocking unrelated UI sections.

HOW:
- useState per responsibility.
- messagesByRepo prevents cross-repository conversation leakage.

CONTEXT:
- This directly enforces RAG workspace boundaries in UI behavior.

CHANGES:
- Replaced old single messages array with messagesByRepo map.


C) Derived state and utility methods
WHAT:
- selectedRepoLabel: fallback display text.
- visibleRepos: filtered repo list by name/namespace.
- activeMessages: current repo conversation slice.
- addToast: pushes toast and auto-removes after timeout.
- appendMessage: appends to one repo only.

WHY:
- Keeps render logic clean and avoids repeated filtering each render path.

HOW:
- useMemo for computed values.
- Functional state updates avoid stale closures.

CONTEXT:
- activeMessages is the key view-layer projection for current repo context.

CHANGES:
- Added filtered sidebar search + toast system.


D) loadRepos workflow
WHAT:
- Calls GET /repos/.
- Sets repos and auto-selects first repo if none active.
- Emits error toast on failure.

WHY:
- Sidebar must reflect server-side ingestion metadata.

HOW:
- Async function called on mount and refresh actions.

CONTEXT:
- Repository list is the context switch source for RAG chat sessions.

CHANGES:
- Previous inline error text replaced by toast notifications.

CHANGES:
- Upgraded from static ingest section to controlled add workspace flow.

CONTEXT:

12) HOW THE NEW FRONTEND SUPPORTS MULTI-REPO RAG
====================================================================

End-to-end flow:
1. User logs in and token is stored.
2. Sidebar loads repositories from /api/repos/.
3. User selects a repo workspace.
4. Chat input sends /api/query/ with repo_name.
5. Response is saved under messagesByRepo[selectedRepo].
6. Switching repos instantly swaps activeMessages without losing previous chats.
7. Ingesting a new repo via /api/ingest adds a new workspace.

Why this is scalable:
- Context separation is explicit in both backend namespace and frontend state map.
- UI remains responsive with memoized derived state and scoped updates.
